System Design for SaaS

In Database Design you designed what the app remembers. This guide zooms out to the whole machine: every box a request passes through, why each exists, and how the architecture grows as a SaaS grows. We'll keep using TeamTasks as the running example — and almost every box here maps to a section of your ProCodrr course.

Part 1: The journey of one request

A user in another city clicks "Add task" in TeamTasks. What actually happens?

Browser
  │ 1. DNS lookup: app.teamtasks.com → an IP          (course Sec 9)
  │ 2. TLS handshake, then HTTPS POST /api/tasks       (Sec 17, 20)
  ▼
NGINX (reverse proxy on your server)                   (Sec 20)
  │ 3. Terminates SSL, forwards to Node on port 3000
  ▼
Express app                                            (Sec 11)
  │ 4. Middleware chain: rate limit → auth → tenant scoping
  │    (Sec 17, 14–15, and Step 8 of the database guide)
  ▼
MongoDB                                                (Sec 12–13)
  │ 5. Insert the task document (with orgId — the tenant key!)
  ▼
Express again
  │ 6. Maybe: invalidate a Redis cache entry           (Sec 16)
  │ 7. Maybe: enqueue a job — "email the assignee"     (Phase 4)
  ▼
Browser gets 201 Created — total round trip: ~100–300 ms

Two big ideas fall out of this trace:

  1. Every box is a separate concern with a separate failure mode. System design is deciding which boxes you need, where they run, and what happens when each one is slow or down.
  2. The database is the only box that remembers. Everything else — NGINX, Express, Redis-as-cache — can be wiped and restarted with no data loss. This is why the database gets its own guide, and why "stateless app servers" (below) is the single most important scaling idea.

The one rule that makes everything else possible: stateless app servers

If Express keeps sessions in its own memory, then user Alice is "logged in" on server 1 but a stranger on server 2 — you can never run two servers. The fix, which your course teaches without naming the principle:

  • Sessions → Redis (Sec 16), not app memory
  • Uploaded files → S3 (Sec 18), not local disk
  • Anything that must survive a restart → MongoDB

Once the app server holds nothing, you can run 1 or 10 of them behind NGINX, kill and replace them freely. This is also what makes zero-downtime deploys (Sec 20) and Lambda (Sec 21) possible — Lambda is just "statelessness taken to the extreme".

Caching: the three places answers can be remembered

  1. Browser/CDN — static assets, cached by URL. Cheap, covered in Sec 18.
  2. Redis — expensive query results ("the task board for project X"), cached with a TTL. The hard part is invalidation: when a task changes, the cached board is now a lie. Pattern: on every write, delete the affected cache keys. Start with no Redis caching and add it only where measurements show pain — a wrong cache is worse than a slow query.
  3. The database's own memory — MongoDB keeps hot data and indexes in RAM automatically. This is why proper indexes (database guide, Step 7) often make Redis caching unnecessary at small scale.

Queues: the request/background split

Rule: a request handler should only do work the user is actively waiting for.

Sending an email takes 1–3 seconds and can fail — if you do it inside POST /api/tasks, every task creation is slow and a mail-provider outage breaks task creation entirely. Instead the handler enqueues a job (milliseconds) and a separate worker process sends the email, with retries. This is the single most common architecture upgrade a course project needs to become production-grade — it's Phase 4 in the roadmap for a reason.

Things to notice:

  • The worker is a separate process running the same codebase (node worker.js next to node server.js) — not a separate project. PM2 (Sec 20) runs both.
  • Stripe webhooks arrive from outside. Billing state changes (payment failed, subscription canceled) come to you — this is why webhook signature verification (Sec 19) matters: that endpoint is publicly reachable.
  • MongoDB Atlas rather than self-managed — backups, upgrades, and monitoring are someone else's 3 a.m. problem (the course's Sec 12 Atlas-vs-self-managed discussion, and roadmap Phase 6 agrees).
  • This whole diagram fits on one $20–40/month of infrastructure. System design for a young SaaS is mostly about arranging well-understood boxes, not inventing exotic ones.

Part 4: How the architecture evolves (and when)

Design for the stage you're at, plus roughly one step of headroom. Not more.

Stage 0 — one box (0 → first users)

Everything from Part 3 on a single EC2 instance / one Render or Railway service. NGINX, Node, Redis, worker, all together; MongoDB on Atlas. This is exactly what Sec 20 teaches. It's not a toy — it comfortably serves thousands of users.

Stage 1 — split by role (traffic grows, deploys get scary)

Move the worker to its own instance so a heavy job can't starve web requests. Use managed Redis. Add a staging environment (roadmap Phase 6). Most SaaS products live happily in this stage for years.

Stage 2 — horizontal scaling (one app server isn't enough)

Run N app servers behind the load balancer. This costs nothing architecturally if you kept servers stateless — that's the payoff. New problem: WebSockets and Socket.IO need Redis pub/sub (Sec 16) as a backplane so a message reaches users connected to any server.

Stage 3 — targeted surgery (real scale, measured bottlenecks)

Read replicas, sharding, extracting one hot service. You are nowhere near this, and that's the point: each stage transition should be triggered by a measured bottleneck (observability, Sec 22 — this is what those dashboards are for), never by imagination.

The anti-pattern to internalize: premature microservices

Splitting TeamTasks into auth-service, task-service, billing-service on day one buys you: network calls where function calls were, distributed transactions where one DB transaction was, and 5 deploy pipelines where one was. It delivers nothing until you have teams that need to ship independently. A well-structured monolith (feature folders — roadmap Phase 3) with a separate worker process is the correct architecture for approximately every solo-founder SaaS. This mirrors mistake #8 in the database guide: designing for imaginary scale.

Part 5: Cross-cutting concerns — a checklist per feature

For any new feature ("file attachments on tasks"), walk the boxes:

  1. Data — new entities/fields? Tenant key present? Indexes for the new queries? (the database guide's whole process)
  2. Request path — new endpoints? Auth + role checks? Rate limits? Input validation?
  3. Slow work — anything over ~500 ms or failure-prone → queue job (virus-scan the upload, generate thumbnails)
  4. Storage — files to S3 via signed URLs (Sec 18), never through your server if avoidable
  5. Caching — does this write invalidate anything cached?
  6. Real-time — should other viewers of the board see it appear live? (Socket.IO — roadmap Phase 4)
  7. Billing — is it plan-gated? ("attachments on Pro only" — entitlements, roadmap Phase 5)
  8. Observability — will you know if it breaks? (error tracking, a metric, an alert)

This checklist is system design, applied at feature scale. Architecture isn't a diagram you draw once; it's this walk, done every time.

Part 6: Common beginner mistakes

  1. State on the app server — in-memory sessions, files on local disk. Breaks the moment you run a second instance or redeploy. (Part 2)
  2. Doing slow work in request handlers — emails, PDFs, video processing inline. One slow dependency takes your whole API down with it. (Part 2, queues)
  3. Caching before measuring — Redis-caching everything on day one, then debugging stale-data bugs for a week. Indexes first, cache later. (Part 2)
  4. Microservices for an app with zero users. (Part 4)
  5. Trusting incoming webhooks — an unverified /stripe-webhook endpoint means anyone can POST "this org paid." Verify signatures. (Part 3, Sec 19)
  6. No worker process — then the first background need gets hacked in with setTimeout, which dies on every deploy. (Part 3)
  7. Scaling the database before adding indexes — 95% of "we need to scale" at small size is a missing index or an N+1 query. (database guide Step 7, roadmap Phase 4)
  8. No staging environment — testing schema migrations for the first time in production. (roadmap Phase 6)

Part 7: Practice path

  1. Trace your course's StorageApp — draw its architecture diagram as it actually is (probably: browser → Express → MongoDB + local disk). Mark every place it violates statelessness. Fix one: move uploads to S3 (Sec 18).
  2. Add a worker — pick any project, add BullMQ + a worker process that sends a (console-logged) email on some event. Kill the worker mid-job and watch the retry happen. This one exercise teaches half of Part 2.
  3. Draw TeamTasks at Stage 0 and Stage 2 — from memory, then compare with Part 3. Write down exactly what changed between the stages and why nothing about the code had to.
  4. Load-test something — run autocannon or k6 (roadmap Phase 4) against a course project. Find the first bottleneck. Is it CPU? A missing index? This turns "scaling" from folklore into measurement.
  5. The capstone, deliberately — before building each capstone feature, run the Part 5 checklist on paper first. That habit is the whole guide.

Tools: excalidraw.com or mermaid flowchart blocks for diagrams; autocannon/k6 for load testing; your Sec 22 Grafana dashboards to watch systems under load.
