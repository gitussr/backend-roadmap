Database Design for SaaS

From WordPress user to database designer: the planning and thinking that happens before you write a single CREATE TABLE or Mongoose schema — then the same design implemented in both MySQL and MongoDB.

1 · WordPress 2 · The design process 3 · In MySQL 4 · In MongoDB 5 · Choosing a database 6 · Common mistakes 7 · Practice path

Part 1: The database you already know (WordPress)

You've been using a professionally designed database for years without knowing it. Open phpMyAdmin on any WordPress site and you'll see tables like:

TableWhat it storesDesign lesson hiding inside
wp_usersOne row per userAn entity table
wp_postsPosts, pages, revisions, menu items — all in one tableOne table reused for many types (a post_type column decides)
wp_postmetaKey–value pairs attached to postsThe EAV pattern — infinitely flexible, painfully slow at scale
wp_terms, wp_term_relationshipsCategories/tags and which posts have themA many-to-many relationship done with a join table
wp_commentsComments, each pointing to a postA one-to-many relationship via comment_post_ID

Two takeaways:

  1. Every column and table exists because someone asked: "What does this app need to remember, and how do the pieces relate?" That question is 90% of database design.
  2. WordPress chose extreme flexibility (wp_postmeta can store anything about anything). The price: querying "all products under $50 in stock" requires joining postmeta to itself multiple times — slow and ugly. Every design is a trade-off. Your job as a designer is choosing trade-offs deliberately instead of inheriting them.

Part 2: The design process (works for MySQL AND MongoDB)

Design is the same regardless of database. Only the final implementation differs. Never start with tables or collections. Start here:

Step 1 — Write down what the app must do (requirements)

Plain sentences. For a SaaS example we'll use throughout — "TeamTasks", a project management SaaS (think a tiny Trello/Asana):

  • People sign up and create an organization (their company)
  • An organization has a subscription plan (free / pro)
  • Users can be invited to an organization with a role (admin / member)
  • Organizations create projects
  • Projects contain tasks
  • Tasks can be assigned to a user, have a status, due date, and comments
  • Tasks can have labels (like "bug", "urgent") — reusable across the project

Step 2 — Find the entities (the nouns)

Underline the nouns in your requirements. Nouns that the app must remember become entities:

User, Organization, Subscription, Project, Task, Comment, Label

Rule of thumb: if it has its own identity and lifespan (can be created, updated, deleted independently), it's an entity. "Status" is not an entity — it's just an attribute of Task. "Role" is not an entity here — it's an attribute of the membership between User and Organization.

Step 3 — Find the relationships (the verbs) and their cardinality

Go back to the sentences. The verbs connecting nouns are relationships. For each, ask two questions in both directions: "One X relates to how many Y?"

RelationshipQuestion both waysCardinality
Organization–SubscriptionAn org has one subscription; a subscription belongs to one org1 : 1
Organization–ProjectAn org has many projects; a project belongs to one org1 : N
Project–TaskA project has many tasks; a task belongs to one project1 : N
Task–CommentA task has many comments; a comment belongs to one task1 : N
User–OrganizationA user can join many orgs; an org has many usersM : N
Task–LabelA task has many labels; a label is on many tasksM : N
User–Task (assignment)A task has one assignee; a user has many assigned tasks1 : N

This table is the heart of your design. Get this right and the schema almost writes itself:

  • 1:1 → merge into one table/document, or separate with a shared key
  • 1:N → the "many" side stores the parent's ID (a foreign key / reference)
  • M:N → in SQL: a join table; in MongoDB: an array of references (usually)

Step 4 — List each entity's attributes

For each entity, list what you must remember about it, and mark what's required, unique, or constrained:

User:         id, name, email (unique, required), passwordHash, createdAt
Organization: id, name, slug (unique), createdAt
Subscription: id, orgId, plan (free|pro), status, currentPeriodEnd, stripeCustomerId
Membership:   userId, orgId, role (admin|member), joinedAt      ← the M:N join!
Project:      id, orgId, name, description, createdBy, createdAt
Task:         id, projectId, title, description, status (todo|doing|done),
              assigneeId (nullable), dueDate (nullable), createdAt
Comment:      id, taskId, authorId, body, createdAt
Label:        id, projectId, name, color
TaskLabel:    taskId, labelId                                   ← the other M:N join

Notice Membership appeared out of nowhere: every M:N relationship in SQL becomes its own table, and that table often grows attributes of its own (role, joinedAt). This is one of the most useful patterns in all of database design — WordPress's wp_term_relationships is exactly this.

Step 5 — Draw it (ER diagram)

Always sketch before you code. Boxes = entities, lines = relationships:

The TeamTasks ER diagram — solid lines are 1:N / 1:1, the dashed line is the Task↔Label many-to-many.

Tools like dbdiagram.io, drawio, or mermaid erDiagram blocks in markdown all work — what matters is sketching at all.

Step 6 — Normalize (SQL brain), then decide what to denormalize

Normalization = removing duplication so every fact lives in exactly one place. The three rules that matter in practice:

  • 1NF — no lists inside a cell. Don't store labels = "bug,urgent" in a column. (That's what join tables are for.)
  • 2NF/3NF — every column must depend on the whole key and nothing but the key. Practical test: if updating one fact requires updating many rows, it's in the wrong place. E.g., don't copy organizationName into every task row — when the org renames, you'd have to update thousands of rows. Store orgId and join.

Why normalize? No update anomalies, no contradictory data.
Why denormalize (sometimes)? Speed — a read that needs 6 joins can be slow. The mature workflow: normalize first, denormalize only when a measured performance problem appears, and document every denormalization because it creates an update burden you now own.

Step 7 — Choose keys, types, constraints, and indexes

  • Primary key: use a surrogate (auto-increment BIGINT or UUID in MySQL; ObjectId in MongoDB) rather than a natural key like email — emails change.
  • Foreign keys (SQL): declare them. They make bad data impossible instead of merely unlikely.
  • Constraints are free tests: NOT NULL, UNIQUE, ENUM/CHECK catch bugs at the database layer even when app code has a hole. (You saw this idea in MongoDB schema validation in your course, Section 12.)
  • Indexes: create one for every column you'll filter or join on (tasks.projectId, memberships.orgId, users.email). Rule of thumb: every foreign key gets an index; every "WHERE" you'll run often gets an index. Indexes cost write speed and storage, so don't index everything.

Step 8 — The SaaS-specific step: multi-tenancy

The single most important question for SaaS: how do you keep customer A from ever seeing customer B's data?

The standard answer for 99% of SaaS apps: shared database, tenant ID column — every row/document that belongs to a customer carries the orgId, and every single query filters by it. In our design, Project has orgId; tasks reach the org through their project. Many teams still stamp orgId directly on Task and Comment too — a deliberate, safe denormalization that lets you query and secure everything by tenant directly. Enforce it in one place (middleware / a base query helper), never by hoping each developer remembers.

Part 3: The same design in MySQL

Now — and only now — implementation. The thinking above translates mechanically:

CREATE TABLE users (
  id            BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  name          VARCHAR(120) NOT NULL,
  email         VARCHAR(255) NOT NULL UNIQUE,
  password_hash VARCHAR(255) NOT NULL,
  created_at    TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE organizations (
  id         BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  name       VARCHAR(120) NOT NULL,
  slug       VARCHAR(60)  NOT NULL UNIQUE,
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE subscriptions (           -- 1:1 with organizations
  id                 BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  org_id             BIGINT UNSIGNED NOT NULL UNIQUE,   -- UNIQUE makes it 1:1
  plan               ENUM('free','pro') NOT NULL DEFAULT 'free',
  status             ENUM('active','past_due','canceled') NOT NULL DEFAULT 'active',
  current_period_end DATETIME NULL,
  stripe_customer_id VARCHAR(64) NULL,
  FOREIGN KEY (org_id) REFERENCES organizations(id) ON DELETE CASCADE
);

CREATE TABLE memberships (             -- the User↔Org M:N join table
  user_id   BIGINT UNSIGNED NOT NULL,
  org_id    BIGINT UNSIGNED NOT NULL,
  role      ENUM('admin','member') NOT NULL DEFAULT 'member',
  joined_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (user_id, org_id),       -- composite key: one row per pair
  FOREIGN KEY (user_id) REFERENCES users(id)         ON DELETE CASCADE,
  FOREIGN KEY (org_id)  REFERENCES organizations(id) ON DELETE CASCADE,
  INDEX idx_org (org_id)               -- "list all members of an org" is common
);

CREATE TABLE projects (
  id          BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  org_id      BIGINT UNSIGNED NOT NULL,  -- the tenant key
  name        VARCHAR(160) NOT NULL,
  description TEXT NULL,
  created_by  BIGINT UNSIGNED NOT NULL,
  created_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (org_id)     REFERENCES organizations(id) ON DELETE CASCADE,
  FOREIGN KEY (created_by) REFERENCES users(id),
  INDEX idx_org (org_id)
);

CREATE TABLE tasks (
  id          BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  project_id  BIGINT UNSIGNED NOT NULL,
  org_id      BIGINT UNSIGNED NOT NULL,  -- deliberate denormalization: tenant safety
  title       VARCHAR(200) NOT NULL,
  description TEXT NULL,
  status      ENUM('todo','doing','done') NOT NULL DEFAULT 'todo',
  assignee_id BIGINT UNSIGNED NULL,      -- nullable: tasks can be unassigned
  due_date    DATE NULL,
  created_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (project_id)  REFERENCES projects(id) ON DELETE CASCADE,
  FOREIGN KEY (org_id)      REFERENCES organizations(id),
  FOREIGN KEY (assignee_id) REFERENCES users(id) ON DELETE SET NULL,
  INDEX idx_project_status (project_id, status),  -- board view: tasks by column
  INDEX idx_assignee (assignee_id)                -- "my tasks" view
);

CREATE TABLE comments (
  id         BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  task_id    BIGINT UNSIGNED NOT NULL,
  author_id  BIGINT UNSIGNED NOT NULL,
  body       TEXT NOT NULL,
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (task_id)   REFERENCES tasks(id) ON DELETE CASCADE,
  FOREIGN KEY (author_id) REFERENCES users(id),
  INDEX idx_task (task_id)
);

CREATE TABLE labels (
  id         BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  project_id BIGINT UNSIGNED NOT NULL,
  name       VARCHAR(40) NOT NULL,
  color      CHAR(7) NOT NULL DEFAULT '#888888',
  FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE,
  UNIQUE KEY uq_label (project_id, name)   -- no duplicate label names per project
);

CREATE TABLE task_labels (                 -- the Task↔Label M:N join table
  task_id  BIGINT UNSIGNED NOT NULL,
  label_id BIGINT UNSIGNED NOT NULL,
  PRIMARY KEY (task_id, label_id),
  FOREIGN KEY (task_id)  REFERENCES tasks(id)  ON DELETE CASCADE,
  FOREIGN KEY (label_id) REFERENCES labels(id) ON DELETE CASCADE
);

Things to study in this schema:

  • ON DELETE CASCADE — delete an org and everything under it vanishes consistently. ON DELETE SET NULL on assignee — deleting a user shouldn't delete their tasks.
  • The composite index (project_id, status) serves the most common query: "show the board for project X" — this is designing indexes from queries, not guessing.
  • Compare with WordPress: task_labels ≈ wp_term_relationships; the difference is we have real typed columns instead of everything-is-postmeta.

Part 4: The same design in MongoDB — different questions

MongoDB design starts from a different place. Instead of "what are the entities?" (you already did that — it's still required), the driving question is:

"What does the application read and write together, most often?"

Because MongoDB lets you embed related data inside a document, every relationship now has a second option:

DecisionEmbed when…Reference when…
Comment inside Task?Data is always read with the parent, is small and boundedData is read on its own, grows unboundedly, or is shared
Label inside Task?—Shared across tasks → reference (embedding would duplicate)
Task inside Project?—Unbounded growth → never embed unbounded arrays

Rules of thumb (from your course's Section 12 "Embedded vs Referenced"):

  1. Embed what you read together. Reference what you share or what grows without limit.
  2. MongoDB has a 16 MB document cap — an embedded array that grows forever is a design bug.
  3. There are no joins enforced by the DB and no foreign key constraints — your app code (Mongoose) is now responsible for integrity that MySQL gave you for free. This is the real trade-off of MongoDB, not "SQL vs NoSQL syntax."

The TeamTasks schema in Mongoose

// users — same as SQL, entities with independent identity stay separate
const userSchema = new Schema({
  name:         { type: String, required: true },
  email:        { type: String, required: true, unique: true, lowercase: true },
  passwordHash: { type: String, required: true },
}, { timestamps: true });

// organizations — subscription is 1:1 and always read with the org → EMBED it
const organizationSchema = new Schema({
  name: { type: String, required: true },
  slug: { type: String, required: true, unique: true },
  subscription: {
    plan:             { type: String, enum: ['free', 'pro'], default: 'free' },
    status:           { type: String, enum: ['active', 'past_due', 'canceled'], default: 'active' },
    currentPeriodEnd: Date,
    stripeCustomerId: String,
  },
}, { timestamps: true });

// memberships — M:N with attributes (role) → its own collection, like the SQL join table
const membershipSchema = new Schema({
  userId: { type: Schema.Types.ObjectId, ref: 'User', required: true },
  orgId:  { type: Schema.Types.ObjectId, ref: 'Organization', required: true },
  role:   { type: String, enum: ['admin', 'member'], default: 'member' },
}, { timestamps: true });
membershipSchema.index({ userId: 1, orgId: 1 }, { unique: true }); // one row per pair
membershipSchema.index({ orgId: 1 });                              // list org members

// projects — labels are small, bounded, project-owned → EMBED them here
const projectSchema = new Schema({
  orgId:       { type: Schema.Types.ObjectId, ref: 'Organization', required: true },
  name:        { type: String, required: true },
  description: String,
  createdBy:   { type: Schema.Types.ObjectId, ref: 'User', required: true },
  labels: [{                       // subdocuments get their own _id automatically
    name:  { type: String, required: true },
    color: { type: String, default: '#888888' },
  }],
}, { timestamps: true });
projectSchema.index({ orgId: 1 });

// tasks — unbounded per project → own collection. Label refs point at the
// embedded subdocument _ids inside project.labels.
const taskSchema = new Schema({
  projectId:  { type: Schema.Types.ObjectId, ref: 'Project', required: true },
  orgId:      { type: Schema.Types.ObjectId, ref: 'Organization', required: true }, // tenant key
  title:      { type: String, required: true },
  description: String,
  status:     { type: String, enum: ['todo', 'doing', 'done'], default: 'todo' },
  assigneeId: { type: Schema.Types.ObjectId, ref: 'User', default: null },
  dueDate:    Date,
  labelIds:   [Schema.Types.ObjectId],   // M:N as an array of refs — no join table!
}, { timestamps: true });
taskSchema.index({ projectId: 1, status: 1 });  // board view
taskSchema.index({ assigneeId: 1 });            // "my tasks"
taskSchema.index({ orgId: 1 });                 // tenant-wide queries

// comments — could embed in task (read together!), but unbounded → own collection.
// If you can cap them ("show latest 20, load more"), embedding the latest few and
// referencing the rest is a known hybrid pattern. Start simple: reference.
const commentSchema = new Schema({
  taskId:   { type: Schema.Types.ObjectId, ref: 'Task', required: true },
  authorId: { type: Schema.Types.ObjectId, ref: 'User', required: true },
  body:     { type: String, required: true },
}, { timestamps: true });
commentSchema.index({ taskId: 1, createdAt: -1 });

What changed between MySQL and MongoDB — and what didn't

MySQLMongoDB
Entities & cardinality analysis (Steps 1–5)identicalidentical
Subscription (1:1)separate table, UNIQUE FKembedded in org document
Labelsown table + join tableembedded in project + array of refs on task
M:N with attributes (membership)join tableseparate collection (same idea!)
IntegrityDB-enforced (FKs, ENUM, CASCADE)app-enforced (Mongoose + discipline)
"Delete an org"one DELETE, cascades handle the restyou write the cascade yourself (or use transactions)
Reading a task boardJOINsone query on tasks + populate() where needed

The planning work is 100% shared. Only Steps 6–7 diverge: SQL normalizes by default and joins at read time; MongoDB pre-joins at write time (embedding) where access patterns justify it.

Part 5: Choosing between MySQL and MongoDB for a SaaS

Honest guidance, not tribalism:

  • Choose MySQL/Postgres when: data is highly relational (SaaS almost always is — orgs, users, plans, invoices), you want integrity enforced by the database, you'll need ad-hoc reporting ("revenue by plan by month"), or a team will grow around it. Most successful SaaS products run on relational databases.
  • Choose MongoDB when: document shapes genuinely vary (CMS content, event payloads, catalogs with wildly different attributes), your access patterns are known and document-shaped, or your stack/team is deeply invested in it (MERN).
  • Either works fine for TeamTasks. At small scale the DB choice matters far less than the design — a well-designed MongoDB schema beats a sloppy MySQL one and vice versa.
  • Whichever you pick, learn the other's mental model anyway. SQL thinking (normalization, joins) makes your MongoDB designs better, and document thinking (access patterns) makes your SQL indexes better.

Part 6: Common beginner mistakes

Each maps to a step above:

  1. Starting from screens instead of data — designing a "dashboard table" instead of entities. Screens change monthly; entities are stable for years. (Step 2)
  2. Storing lists in a field — "bug,urgent" in a column, or in MongoDB, embedding an unbounded array. (Step 6, Part 4)
  3. Missing join tables — trying to model M:N with a column on one side. (Step 3)
  4. Natural primary keys — using email/username as the key; they change. (Step 7)
  5. No indexes on foreign keys / filter columns — works fine with 100 rows, dies at 100,000. (Step 7)
  6. Copying data without owning the update path — denormalizing userName into comments, then user renames and old comments show stale names (sometimes that's even desired — decide consciously!). (Step 6)
  7. Forgetting the tenant filter — the classic SaaS data-leak bug. Centralize it. (Step 8)
  8. Designing for imaginary scale — sharding plans for an app with 0 users. Design cleanly, index sensibly, ship, measure.

Part 7: Practice path

Do these in order; each takes an evening or two. For every one: requirements → entities → cardinality table → ER diagram → both MySQL DDL and Mongoose schemas.

  1. Blog with categories and comments — redesign WordPress's core yourself, then compare your answer to the real wp_* tables. Where did you do better? Where did they, and why?
  2. URL shortener with click analytics — forces you to think about a high-write table (clicks) and aggregation.
  3. The course's StorageApp — you're building it anyway; reverse-engineer its ER diagram from the code and critique it.
  4. E-commerce store — products with variants (size/color), carts, orders, order line items. Line items teach "snapshot" design: an order must record the price at purchase time, a deliberate denormalization.
  5. TeamTasks from this guide — build it for real as your Phase 7 SaaS. Add: activity feed, file attachments, and a "move task between projects" feature — each will stress-test the design.

Tools: dbdiagram.io (ER diagrams from text), MySQL Workbench (visual schema design), MongoDB Compass (inspect real documents), and mermaid erDiagram blocks in markdown (versionable diagrams).
